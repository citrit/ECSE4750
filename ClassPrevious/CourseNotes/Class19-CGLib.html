The Computer Graphics Window library:

The Window:

  • A component to handle Windows and events for our system (a component may be a collection of several classes)
  • As we have seen it should handle:
    1. initialization of the window system
    2. Setting of attributes
    3. Drawing methods
    4. Callbacks for additional drawing control and animation.
  • The Source: cgxwindow.cpp  cgxwindow.h

Geoemtry:

  • We will be using STL vectors for our collections. The first one we find is the Coordinates.
  • The source:   cgcoords.h (aren't STL containers nice, no object code)
  • The nice thing about vectors are they contiguous memory blocks allocated to hold our array.
  • This both minimizes access time by allowing the lookups to be simple offsets
  • Vectors also minimize memory requirements by not requiring extra information per block.

Transformations:

  • But what about what we did in class?
  • The important parts are:
    1. W can apply each of our transforms (rotate, sclae, and translate) into indivdual matrices
    2. We need to concatenate our operations into a single matrix.
    3. Order is important.
  • These classes rely on 3x3 matrices to hold our transforms and apply the resultant composite matrix on the points.
  • Here is an implementation of the Transform class and accompanying Matrix3x3 class: cgtransform.h   cgtransform.cpp
  • The transform class is a convenience class meant to expose easy to use methods required for geometry transformation.
  • It does not restrict your usage to a specific order so we are free to organize and schedule our transforms in any order we choose.
  • This of course makes life harder for the programmer.
  • Any one think of a way to force the user to do the "right thing"?

One Implementation:

  • Here is a procedural way to do things: animate.cpp
  • This relies on the programmer to create a structure to hold the geoemtry and call specific code to draw the geoemtry at every time step.
  • What we want is to free the programmer fromt he everyday toils of arrays and structures and what order are the points.
  • The thing we need is tolpology holders.
  • We have the basic building block, the Shape class.

A More Object Oriented Approach:

  • We are using a collection based system so things are organized this way:
    • The CGWindow (cgxwindow.h, cgxwindow.cpp) holds a collection of CGActors.
    • The CGActor (cgactor.h, cgactor.cpp) holds a collection of shapes.
    • Shapes, our topology containers, hold pointers to the geoemtry (cgcoords.h). Here is an example of a PointShape (cgpointshape.h, cgpointshape.cpp)
  • The CGActors are responible for position and orientation of their CGShapes using a member variable CGTransform (cgtransform.h, cgtransform.cpp)
  • This transform is used when the shapes attempt to draw themselves, for instance the PointShape will use the supplied transform to orient its point prior to calling the CGWindows draw methods.
  • Here is an example of this appoach: main.cpp
