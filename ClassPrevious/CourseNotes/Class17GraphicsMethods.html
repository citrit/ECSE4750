From Windows to drawing, how do we get organized.

Basic Requirements

  • We investigated windows and events to see whats necessary for an application to survive and operate correctly
    • Initialization
    • Event handlers, Input/Output
    • Drawing canvas
    • Menus
  • We investigated basic algorithms for drawing primitives.
    • Lines, points, polygons, etc.
    • Optimizations and implementations.
  • We now need to transition from our procedural methodology to a more object oriented approach.
  • We do this to solve several problems:
    1. (I have to put this) we want re-usability by creating components.
    2. We need to have dynamic creation and removal of entities. Objects allow us to implicitly represent our objects and encapsulate their variables and support objects.
    3. We need to share as much data as possible in order to minimize our memory requirements.
    4. Each object needs to act independant of all others unless part of a larger assembly.

Generating a design

  • Now with our understanding of how to map geometric primitives to the screen, we need methods by which we can efficiently store the necessary information to recreate large geometric representations of our scenes.
  • The choice of storage methods is important because it affects our ability to interface to external data and the overall performance of the entire graphics system. We will define these as our design criterion:
    Compact
    Since we know graphics data tends to be very large, we need to design our structures to minimize computer memory requirements. 
    Efficient
    Scene data needs to be accessed many times and we would like to retrieve and store data in our structures in constant time. 
    Mappable
    Data representations need to efficiently map into graphics primitives. We must be able to easily convert external data into internal graphics data structures. 
    Minimal Coverage
    We must create a data representation that minimizes the number of data types while covering a large portion of external data types. We wish to balance efficiency with the number of data types. 
    Simple
    Simple algorithms and data sets are easier to implement and also easier to extend. 

Graphics Primitives

  • We need to define a base set of primitives we will support that ensures our ability to draw any shape object.
  • These can be defined by:

    Vertex Line PolyLine
    Triangle TriangleStrip Polygon
  • This is a short list and I am sure you can think of others, but for now this will do.
  • We will be using the traditional OO design of creating a base shape class and deriving from this.
  • Our base class needs at least the method to Draw itself.
  • We will add more as things come about.
  • The Shape object is an abstract form which simply defines the concept of the behavior of the concrete derived classes to follow.
  • The abstraction consists of two sections: the Structure and Attributes.
  • Structure consist of two elements: Topology and Geometry
    1. Topology is the information about the structure of the Shape object.
    2. Geometry is the instantiation of the Topology
  • The best way to think of it is if we say a polygon is a triangle, we specified the topology. When we provide the point coordinates we specify the geometry.
  • Attributes are supplemental information which we will call color, texture, line style, etc.
  • So now we have a Window to display our graphics, we have primitives to handle storage, how about scene organization.

Specifying the Scene

  • There a a few options available:
    1. First we can look into using a Scene Graph.
      • Open Inventor is an object-oriented toolkit for developing interactive, 3D graphics applications. It also defines a standard file format for exchanging 3D data among applications.
      • Scene graphs describe geometry and attributes
      • They use a graph (usually a directed acyclic graph) to relate/bind attributes and geometry.
      • Graphical objects (nodes) are organized into a hierarchical graph
      • The path to any node on the graph defines the attributes of that node
      • For example a Rotation or Material node will affect the nodes after it in the scene
      • Strengths:
        • Reuse of properties and transformations reduce storage and rendering times
        • State based drawing paradigm is very fast
        • Hierarchical model allows grouping of nodes
        • It can be part of a larger OO package
      • Weakness:
        • It is a recursive traversal model, overhead involved - pushing & popping the execution state
        • Geometry nodes are effected by “unknown” nodes prior in the tree
        • Cannot render "out of order", always a left to right rendering order
          • Cannot perform parrallel rendering
        • It is difficult to understand, great deal of nuance
      • Scene Graphs serve best as scene description formats
    2. The second option is a Collection based system.
      • A collection based system works on the principal that there are data sources and sinks.
      • The sinks are the initializers of the system in that they signal the other objects to do an update.
      • Each object is responsible for making sure they are up to date and if necessary, recalculate themselves.
      • This helps solve some issues: we can cache data at various levels based on current modification time, we can share data between objects.
